# 006 — MCP over OAuth

**Status:** Implemented

How Fridge is exposed as an OAuth-protected MCP server. The server (Streamable HTTP
+ OAuth 2.1) is built and tested; see "Phases".

## Goal

Serve MCP over **Streamable HTTP**, integrated into the same Hono process as the
web UI (one deployment, one auth layer — see [001-techstack.md](./001-techstack.md)),
and protect it with **OAuth 2.1** so public MCP clients (e.g. Claude) can connect.

## Roles

Per the MCP authorization spec, an MCP server is an OAuth 2.1 **Resource Server**
and the client is an OAuth 2.1 **client**. Fridge will be **both the Resource
Server and the Authorization Server** (self-contained, no external IdP): the
existing name/password login (see [002-user.md](./002-user.md)) authenticates the
user during the authorization step, and the app issues tokens to MCP clients.

## What the spec requires

- **Protected Resource Metadata** (RFC 9728) at
  `/.well-known/oauth-protected-resource`, pointing at the authorization server.
- **Authorization Server Metadata** (RFC 8414) at
  `/.well-known/oauth-authorization-server`.
- **Authorization Code + PKCE** (OAuth 2.1); public clients, no client secret.
- **Dynamic Client Registration** (RFC 7591) at `/register` — MCP clients register
  themselves.
- **Resource Indicators** (RFC 8707): the client sends `resource` on authorize and
  token requests; tokens are audience-bound to the MCP server's canonical URI, which
  the server validates.
- `/mcp` requires a valid `Bearer` access token; a `401` carries
  `WWW-Authenticate` pointing back to the resource metadata.

## Design choices

- **Transport:** `@hono/mcp` (`StreamableHTTPTransport`) + `@modelcontextprotocol/sdk`.
- **Tokens:** opaque tokens stored in Postgres (revocable, auditable; fits the
  DB-centric design) rather than self-encoded JWTs.
- **Base URL:** OAuth metadata must advertise absolute URLs, so the issuer/base URL
  is configurable per environment (localhost / Render) via an env var
  (`PUBLIC_BASE_URL`, falling back to `RENDER_EXTERNAL_URL`).

## Phases

1. **MCP endpoint, no auth.** ✅ Done. `/mcp` (Streamable HTTP, stateless) is mounted
   via `@hono/mcp` in `src/mcp/`, with a `ping` tool.
2. **OAuth (AS + RS).** ✅ Done. `src/routes/oauth/provider.ts` is a Postgres-backed
   `OAuthServerProvider` (opaque tokens stored as SHA-256 hashes). Discovery
   metadata is served at the well-known root (`wellKnownRouter`, per spec), while
   the operation endpoints are mounted under **`/oauth`** (`/oauth/authorize`,
   `/oauth/token`, `/oauth/register`, `/oauth/revoke`) using the individual
   `@hono/mcp/auth` handlers — clients reach them via the discovered metadata.
   `/authorize` reuses the login session (single admin → auto-approve, no consent
   screen); `/mcp` is Bearer-protected (`src/middlewares/oauth.ts`). `tests/mcp/` drives
   the whole flow (discover → DCR → authorize → token → MCP call) over real HTTP,
   so it runs in CI.

## Tools

Each tool is one file under `src/mcp/` (pantry tools grouped in `src/mcp/pantry/`),
registered via `loggedTool` and wired into `buildMcpServer` (`src/mcp/index.ts`).
User-scoped tools receive the caller's
`userId`, which `requireBearer` resolves from the Bearer token (`info.extra.userId`)
and stashes on `c.var.userId` — so a tool only ever touches the caller's own data.

- `ping` — health check, echoes a message.
- `list_pantry` — list the caller's pantry items, each with a computed expiry date.
  Defaults to in-stock only; pass `inStockOnly: false` to include consumed items.
- `add_pantry_item` — add an item (`name`, optional `stockDate`, `bestBeforeDays`,
  `status`).
- `update_pantry_item` — partial update by `id`; omitted fields are left unchanged.
- `consume_pantry_item` — mark an item consumed by `id`.
- `remove_pantry_item` — delete an item by `id`.

The pantry tools delegate to the `PantryItem` model (`src/models/pantry.ts`), the
same data access the web routes use. Each has a test in `tests/mcp/` driving it over
the real OAuth + HTTP flow.

## Local testing

- **Automated:** `npm run test:mcp` runs the full flow (discover → DCR → authorize →
  token → MCP call) over real HTTP.
- **Interactive:** the MCP Inspector (`@modelcontextprotocol/inspector`) walks the
  OAuth flow in a browser against `http://localhost:3000`.

## Gotchas (learned the hard way)

- **Don't advertise scopes you don't grant.** If the metadata lists
  `scopes_supported`, clients (e.g. ChatGPT) will request those scopes at
  `/authorize`, and the handler rejects any scope the client wasn't *registered*
  with → `invalid_scope: Client was not registered with scope <x>`. A DCR client
  registers itself with no scopes, so advertising `["mcp"]` broke ChatGPT. We model
  no scopes, so we advertise none. If you add scopes later, also register clients
  with them (in `registerClient`).
- **Render build cache can serve stale code.** A `render deploys create --commit
  <sha>` once kept serving the previous build (the metadata fix wasn't live even
  though the deploy showed the new commit). If a code change doesn't take effect,
  redeploy with **Clear build cache**. (Render builds with `npm install`, and the
  fix is the dashboard's "Clear build cache & deploy".)
- **Test scope ≠ client scope.** The vitest smoke flow didn't send a `scope`, so it
  passed while ChatGPT (which does) failed. The regression test now asserts the
  metadata advertises no scopes.
